01

Kernel Fusion

The Problem

In early vLLM implementations, attention computation was split across multiple kernels: QK^T matmul, softmax, and attention-V matmul. Each kernel wrote intermediate results to HBM (High Bandwidth Memory). For Llama-70B with long contexts, profiler showed HBM bandwidth saturated at 900 GB/s while GPU compute sat at only 40% utilization. The bottleneck: memory round-trips, not compute.

The Solution

Technique: Flash Attention-style kernel fusion

Reference: vLLM PR #1456

# Before: 3 separate kernels
q_k = torch.matmul(q, k.transpose(-2, -1))  # writes N×N tensor to HBM
attn = torch.softmax(q_k / sqrt(d), dim=-1)  # reads from HBM, writes back
output = torch.matmul(attn, v)               # reads from HBM again

# After: Fused Flash Attention kernel
output = flash_attn_func(q, k, v, causal=True)
# All intermediates stay in SRAM (on-chip memory)
# Result: 2.3x speedup for prefill, HBM traffic reduced by 5-8x

The Pattern

✅ When to Apply

  • Multiple small kernels writing intermediate tensors to HBM
  • Profiler shows bandwidth-bound (high DRAM throughput, low compute utilization)
  • Intermediate results are only used once and can fit in SRAM

📋 How to Apply

  • Identify kernel chains with intermediate HBM writes
  • Fuse into single kernel keeping intermediates in registers/shared memory
  • Use tiling if intermediate data exceeds SRAM capacity

🔍 Profiler Signals

  • DRAM throughput > 70% peak bandwidth
  • Compute utilization < 50%
  • Many small kernel launches (<100μs each)

Memory Access Coalescing

The Problem

vLLM's PagedAttention manages KV cache in non-contiguous memory blocks (like OS virtual memory). Early implementations used naive indexing: each thread accessed KV blocks with strided patterns, causing uncoalesced memory transactions. On A100, this resulted in only 300 GB/s effective bandwidth versus theoretical 2000 GB/s—an 85% efficiency loss.

The Solution

Technique: Coalesced memory access via block reordering

Reference: vLLM PR #891

# Before: Strided access (bad)
@triton.jit
def paged_attention_kernel(...):
    block_idx = tl.program_id(0)
    offset = block_idx * BLOCK_SIZE
    # Each thread accesses different blocks - strided pattern
    kv_data = tl.load(kv_cache_ptr + offset + thread_id * stride)

# After: Coalesced access (good)
@triton.jit
def paged_attention_kernel(...):
    block_idx = tl.program_id(0)
    # All threads in warp load contiguous block
    block_offset = block_idx * BLOCK_SIZE
    kv_data = tl.load(kv_cache_ptr + block_offset + tl.arange(0, BLOCK_SIZE))
    # Adjacent threads load adjacent addresses
# Result: 1.8x speedup, memory efficiency: 300 GB/s → 1600 GB/s

The Pattern

✅ When to Apply

  • Memory-bound kernels with low effective bandwidth
  • Profiler shows many uncoalesced transactions
  • Strided or scattered memory access patterns

📋 How to Apply

  • Ensure adjacent threads access adjacent memory addresses
  • Use vectorized loads (load4, load128, etc.)
  • Reorder data layout if needed (AoS → SoA)

🔍 Profiler Signals

  • High ratio of memory transactions to bytes loaded
  • Nsight: "Global Load Efficiency" < 50%
  • Many L1/L2 cache misses

FP8 Quantization

The Problem

During decode phase in vLLM, each new token requires loading the entire KV cache (all previous keys and values) from HBM to compute attention. For Llama-70B with 4096-token context, this meant transferring 280 GB of FP16 data per batch. With 50 requests in flight, the GPU spent 95% of time waiting on memory, achieving only 12 tokens/sec throughput.

The Solution

Technique: FP8 KV cache with dynamic quantization

Reference: vLLM PR #3005

# Before: FP16 KV cache
class PagedAttention:
    def forward(self, q, k_cache, v_cache):
        # k_cache, v_cache are FP16 (2 bytes per element)
        attn = torch.matmul(q, k_cache.transpose(-2, -1))
        out = torch.matmul(attn, v_cache)

# After: FP8 KV cache with on-the-fly dequant
class PagedAttentionFP8:
    def forward(self, q, k_cache_fp8, v_cache_fp8, k_scale, v_scale):
        # k_cache_fp8 is FP8 (1 byte per element) - 50% memory saved
        k_cache = k_cache_fp8.to(torch.float16) * k_scale  # Dequant in register
        v_cache = v_cache_fp8.to(torch.float16) * v_scale
        attn = torch.matmul(q, k_cache.transpose(-2, -1))
        out = torch.matmul(attn, v_cache)
# Result: 1.9x decode speedup, 50% memory bandwidth saved, <0.1% accuracy loss

The Pattern

✅ When to Apply

  • Memory bandwidth bottleneck (loading large tensors repeatedly)
  • Tensors with low reuse (loaded once, used immediately)
  • FP32/FP16 precision not critical for specific tensor

📋 How to Apply

  • Store data in lower precision (FP8, INT8)
  • Dequantize on-chip during computation (registers or SRAM)
  • Use per-tensor or per-channel scaling factors
  • Profile accuracy impact before deploying

🔍 Profiler Signals

  • Large tensors dominating HBM traffic
  • Low arithmetic intensity (FLOPS/byte ratio)
  • Decode-phase bottleneck (not prefill)

Prefix Caching

The Problem

In chat applications, many requests share common system prompts (e.g., "You are a helpful assistant..."). vLLM was recomputing attention for identical prompt prefixes across different requests. For a 512-token system prompt across 100 requests, this wasted 51,200 token-worth of prefill compute—taking 8 seconds on A100 with no benefit.

The Solution

Technique: Content-addressable KV cache blocks

Reference: vLLM PR #3687

# Before: Recompute everything
class KVCacheManager:
    def process_request(self, prompt_tokens):
        # Always compute from scratch
        for token in prompt_tokens:
            k, v = model.compute_kv(token)
            kv_cache.append(k, v)

# After: Hash-based cache lookup
class KVCacheManager:
    def __init__(self):
        self.block_hash_table = {}  # hash → KV block mapping

    def process_request(self, prompt_tokens):
        # Hash prompt prefix
        prefix_hash = hash(tuple(prompt_tokens[:common_prefix_len]))

        if prefix_hash in self.block_hash_table:
            # Reuse cached KV blocks (zero compute!)
            kv_cache = self.block_hash_table[prefix_hash].copy()
            start_idx = common_prefix_len
        else:
            kv_cache = []
            start_idx = 0

        # Only compute new tokens
        for token in prompt_tokens[start_idx:]:
            k, v = model.compute_kv(token)
            kv_cache.append(k, v)
# Result: 5-10x speedup for requests with shared prefixes, zero accuracy loss

The Pattern

✅ When to Apply

  • Repeated computation across different inputs
  • Deterministic operations (same input → same output)
  • High cost of recomputation vs. cache lookup

📋 How to Apply

  • Identify common input patterns (shared prefixes, repeated sequences)
  • Use content-addressable storage (hash table keyed by input)
  • Implement cache eviction policy (LRU for memory limits)
  • Ensure determinism (no dropout, temperature=0 for KV)

🔍 Profiler Signals

  • High prefill latency relative to decode
  • Many requests with identical prompt prefixes
  • Compute time dominated by redundant work

Block Size Tuning

The Problem

Triton attention kernels use block size (BLOCK_M, BLOCK_N) to tile the attention computation. Using fixed blocks (e.g., BLOCK_M=128, BLOCK_N=64) works well for common sequence lengths but fails at extremes. For short sequences (128 tokens), large blocks leave 75% of threads idle. For long sequences (8192 tokens), small blocks cause 10x more kernel launches and synchronization overhead.

The Solution

Technique: Autotuning with heuristics

Reference: Triton's autotune decorator and vLLM's kernel selection

# Before: Fixed block size
@triton.jit
def attention_kernel(Q, K, V, Out,
                     BLOCK_M: tl.constexpr = 128,
                     BLOCK_N: tl.constexpr = 64):
    # Always uses 128x64 blocks regardless of sequence length
    ...

# After: Autotuned configs
@triton.autotune(
    configs=[
        triton.Config({'BLOCK_M': 128, 'BLOCK_N': 128}, num_warps=8),  # Long seqs
        triton.Config({'BLOCK_M': 64, 'BLOCK_N': 64}, num_warps=4),    # Medium
        triton.Config({'BLOCK_M': 32, 'BLOCK_N': 32}, num_warps=2),    # Short seqs
    ],
    key=['seq_len'],  # Tune based on sequence length
)
@triton.jit
def attention_kernel(Q, K, V, Out, seq_len,
                     BLOCK_M: tl.constexpr,
                     BLOCK_N: tl.constexpr):
    # Triton benchmarks all configs and picks best for seq_len
    ...

# Runtime selection based on heuristics
def select_kernel(seq_len):
    if seq_len < 512:
        return attention_kernel_small  # BLOCK_M=32
    elif seq_len < 2048:
        return attention_kernel_medium  # BLOCK_M=64
    else:
        return attention_kernel_large  # BLOCK_M=128
# Result: 1.4-2.1x speedup across varied sequence lengths

The Pattern

✅ When to Apply

  • Performance varies significantly with input size/shape
  • Trade-off between occupancy and efficiency
  • Multiple valid configurations exist

📋 How to Apply

  • Define 3-5 candidate configurations covering input range
  • Use Triton's @autotune or manual benchmarking
  • Optionally add heuristics for runtime selection
  • Consider occupancy (small blocks) vs. efficiency (large blocks)

🔍 Profiler Signals

  • Low occupancy (<50%) on some inputs
  • Performance cliff at certain input sizes
  • Many idle warps or underutilized SMs

Static Shape Specialization

The Problem

PyTorch's eager mode uses dynamic dispatch: every operation checks tensor shapes at runtime, looks up the appropriate kernel, and handles general cases (broadcasting, strided tensors). For a simple MLP layer called 1000 times per forward pass, this added 15% overhead—120ms of pure Python/dispatch time on top of 680ms compute.

The Solution

Technique: torch.compile with static shapes

Reference: PyTorch 2.0 torch.compile documentation

# Before: Dynamic dispatch (general but slow)
class MLP(nn.Module):
    def forward(self, x):
        # Every call: check x.shape, dispatch to appropriate kernel
        x = F.linear(x, self.w1, self.b1)  # Dynamic shape check
        x = F.gelu(x)                       # Dynamic shape check
        x = F.linear(x, self.w2, self.b2)  # Dynamic shape check
        return x

model = MLP()
# Runtime: 800ms (680ms compute + 120ms dispatch overhead)

# After: Compiled with static shapes
model = MLP()
model = torch.compile(model, mode="max-autotune")

# First call traces the model and generates optimized code
with torch.no_grad():
    output = model(input)  # shape: [batch=32, hidden=4096]

# PyTorch generates specialized kernel assuming:
# - batch_size=32, hidden=4096 (static, no runtime checks)
# - Fuses linear + gelu + linear into single kernel
# - Removes all Python overhead

# Subsequent calls with same shape:
# Runtime: 680ms (pure compute, zero dispatch overhead)
# Result: 1.18x speedup by eliminating dynamic checks

The Pattern

✅ When to Apply

  • Hot path with many small operations
  • Fixed input shapes across many iterations
  • Profile shows Python/dispatch overhead >10%

📋 How to Apply

  • Use torch.compile (PyTorch 2.0+) or TorchScript
  • Specify static shapes via dynamic=False or constraints
  • Run warm-up iterations to trigger compilation
  • For dynamic batches, compile multiple batch sizes

🔍 Profiler Signals

  • High Python overhead in profiler (aten::record_function)
  • Many small kernel launches (<50μs each)
  • Low GPU utilization despite high throughput demand

Asynchronous Memory Copy

The Problem

In multi-GPU tensor parallelism, AllReduce operations copy gradients across GPUs synchronously—blocking compute until the transfer completes. For Llama-70B across 8 GPUs, each layer's AllReduce took 12ms while compute was idle, adding 960ms overhead for 80 layers.

The Solution

Technique: Overlap communication with computation using CUDA streams

Reference: PyTorch distributed communication optimizations

# Before: Synchronous (blocking)
def forward_layer(x, layer):
    y = layer(x)  # Compute
    torch.distributed.all_reduce(y)  # Wait for communication to complete
    return y

# After: Async with overlap
def forward_layer(x, layer):
    # Split into chunks
    y = layer(x)
    chunks = y.chunk(4, dim=0)

    # Start communication for chunk 0 in separate stream
    comm_stream = torch.cuda.Stream()
    with torch.cuda.stream(comm_stream):
        torch.distributed.all_reduce(chunks[0], async_op=True)

    # Compute next chunks while chunk 0 is transferring
    for i in range(1, 4):
        compute_chunk(chunks[i])

    # Sync streams
    torch.cuda.current_stream().wait_stream(comm_stream)
    return torch.cat(chunks, dim=0)
# Result: 30-40% reduction in communication overhead

The Pattern

✅ When to Apply

  • Communication-heavy workloads (multi-GPU, distributed training)
  • Independent compute and communication operations
  • Profiler shows GPU idle during transfers

📋 How to Apply

  • Use CUDA streams to run communication concurrently
  • Split data into chunks (communicate chunk N while computing chunk N+1)
  • Insert synchronization points only when needed

🔍 Profiler Signals

  • High communication time in distributed training
  • GPU idle during AllReduce/AllGather
  • Timeline shows sequential compute → transfer pattern

Shared Memory Bank Conflicts

The Problem

GPU shared memory is divided into 32 banks. When multiple threads in a warp access the same bank simultaneously (but different addresses), accesses serialize—turning a 1-cycle operation into a 32-cycle operation. Matrix transpose kernels commonly hit this, reducing effective bandwidth by 16-32x.

The Solution

Technique: Padding to avoid bank conflicts

Reference: CUDA matrix transpose optimizations

# Before: Bank conflicts
@triton.jit
def transpose_kernel(...):
    # Allocate shared memory tile: 32x32 elements
    tile = tl.zeros((32, 32), dtype=tl.float16)

    # Load from global memory (no conflicts)
    tile = tl.load(input_ptr + offsets)

    # Transpose: thread i writes to tile[i, j], reads from tile[j, i]
    # Problem: threads 0-31 all access bank (j % 32) → 32-way conflict!
    transposed = tl.trans(tile)
    tl.store(output_ptr + offsets, transposed)

# After: Padded shared memory
@triton.jit
def transpose_kernel(...):
    # Add 1 element padding: 32x33 instead of 32x32
    # This shifts subsequent rows to different banks
    tile = tl.zeros((32, 33), dtype=tl.float16)

    tile[:, :32] = tl.load(input_ptr + offsets)

    # Now threads access different banks (offset by padding)
    transposed = tl.trans(tile[:, :32])
    tl.store(output_ptr + offsets, transposed)
# Result: 8-16x speedup for transpose operations

The Pattern

✅ When to Apply

  • Kernels using shared memory for staging/reuse
  • Transpose, reduction, or shuffle operations
  • Profiler shows "Shared Memory Replay" warnings

📋 How to Apply

  • Add padding to shared memory arrays (e.g., [32][32] → [32][33])
  • Use Nsight Compute to measure bank conflicts
  • Alternative: Reorder accesses to avoid conflicts

🔍 Profiler Signals

  • Nsight: "Shared Memory Bank Conflicts" > 0
  • Low shared memory bandwidth utilization
  • Transpose/reduction kernels underperforming

Warp Shuffle Operations

The Problem

Reduction operations (sum, max) traditionally use shared memory: each thread writes its value, synchronizes, then reads neighbors' values. For warp-level reductions (32 threads), this shared memory round-trip adds 20-30 cycles and wastes precious SRAM capacity.

The Solution

Technique: Warp shuffle instructions for register-only communication

Reference: CUDA warp-level primitives (intrinsics)

# Before: Shared memory reduction
@triton.jit
def warp_reduce_sum(val):
    tid = tl.program_id(0)
    shared = tl.alloc(32, dtype=tl.float32)  # Allocate shared memory
    shared[tid] = val
    tl.barrier()  # Synchronize

    # Tree reduction
    if tid < 16: shared[tid] += shared[tid + 16]
    tl.barrier()
    if tid < 8: shared[tid] += shared[tid + 8]
    tl.barrier()
    # ... more barriers
    return shared[0]

# After: Warp shuffle (register-only)
@triton.jit
def warp_reduce_sum(val):
    # No shared memory, no barriers!
    # Each thread shuffles data directly from other threads' registers
    for offset in [16, 8, 4, 2, 1]:
        val += tl.shfl_down(val, offset)  # Get value from thread (tid + offset)
    return val  # Thread 0 has final sum
# Result: 4-6x faster, zero shared memory usage

The Pattern

✅ When to Apply

  • Warp-level reductions (sum, max, any/all)
  • Broadcasting within a warp
  • Shared memory pressure or latency issues

📋 How to Apply

  • Use warp shuffle intrinsics (shfl_down, shfl_xor, shfl_sync)
  • No synchronization needed (implicit within warp)
  • Works for any warp-local communication pattern

🔍 Profiler Signals

  • High shared memory usage from reductions
  • Many __syncthreads() calls
  • Small reduction kernels (<32 threads per block)

Loop Unrolling

The Problem

Tight inner loops with small iteration counts spend significant time on loop overhead: incrementing counters, checking conditions, branching. For a 16-iteration loop doing simple arithmetic, branch mispredictions and pipeline stalls added 30% overhead.

The Solution

Technique: Manual or compiler-directed loop unrolling

Reference: Compiler optimization techniques

# Before: Tight loop
@triton.jit
def dot_product(a, b):
    result = 0.0
    for i in range(16):  # 16 iterations, 16 branches
        result += a[i] * b[i]
    return result

# After: Unrolled (4x)
@triton.jit
def dot_product(a, b):
    result = 0.0
    # Unroll by factor of 4: 4 iterations, 4 branches
    for i in range(0, 16, 4):
        result += a[i] * b[i]
        result += a[i+1] * b[i+1]
        result += a[i+2] * b[i+2]
        result += a[i+3] * b[i+3]
    return result

# Or: Fully unrolled (no loop at all)
@triton.jit
def dot_product(a, b):
    # Triton often does this automatically for small constexpr loops
    result = (a[0]*b[0] + a[1]*b[1] + a[2]*b[2] + a[3]*b[3] +
              a[4]*b[4] + a[5]*b[5] + a[6]*b[6] + a[7]*b[7] +
              a[8]*b[8] + a[9]*b[9] + a[10]*b[10] + a[11]*b[11] +
              a[12]*b[12] + a[13]*b[13] + a[14]*b[14] + a[15]*b[15])
    return result
# Result: 20-40% speedup for small loops, better instruction pipelining

The Pattern

✅ When to Apply

  • Inner loops with small, known iteration counts (<32)
  • Simple loop bodies (few instructions per iteration)
  • Profile shows time spent in loop overhead

📋 How to Apply

  • Use #pragma unroll in CUDA or tl.constexpr loops in Triton
  • Unroll factor: 2-8x depending on register pressure
  • Fully unroll very small loops (<16 iterations)

🔍 Profiler Signals

  • High branch instruction count
  • Pipeline stalls from control flow
  • Small loop bodies dominating execution time

Vectorized Load/Store

The Problem

Loading data one element at a time (float16, 2 bytes) wastes memory transaction capacity. GPUs can fetch 128 bytes per transaction, but scalar loads only use 2 bytes—a 98% efficiency loss. Copying a 1GB buffer took 18ms when theoretical minimum was 0.5ms.

The Solution

Technique: Vectorized loads (load multiple elements per instruction)

Reference: CUDA vector types (float4, uint4)

# Before: Scalar loads
@triton.jit
def copy_kernel(src, dst):
    tid = tl.program_id(0)
    # Load 1 element at a time (2 bytes for float16)
    for i in range(1024):
        dst[tid * 1024 + i] = tl.load(src + tid * 1024 + i)
# Bandwidth: 50 GB/s (2% efficiency)

# After: Vectorized loads
@triton.jit
def copy_kernel(src, dst):
    tid = tl.program_id(0)
    # Load 8 elements at a time (16 bytes = 8 x float16)
    # Triton automatically vectorizes with tl.load on ranges
    for i in range(0, 1024, 8):
        vec = tl.load(src + tid * 1024 + i + tl.arange(0, 8))
        tl.store(dst + tid * 1024 + i + tl.arange(0, 8), vec)
# Bandwidth: 1800 GB/s (90% efficiency)

# Or use explicit vectorization in CUDA:
# float4* src_vec = (float4*)src;  // Cast to vector type
# float4 data = src_vec[tid];      // Load 16 bytes in one instruction
# Result: 8-16x bandwidth improvement

The Pattern

✅ When to Apply

  • Memory-bound kernels with low bandwidth utilization
  • Contiguous memory accesses (no gaps/strides)
  • Element size < 16 bytes (room for vectorization)

📋 How to Apply

  • Load/store 2-8 elements at once (float2, float4, float8)
  • Ensure alignment (addresses must be multiples of vector size)
  • Triton auto-vectorizes tl.load/store on ranges

🔍 Profiler Signals

  • Low "Global Load Throughput" (<20% theoretical)
  • Many small transactions (8-32 bytes each)
  • Memory-bound with underutilized bus

Split-K GEMM

The Problem

Matrix multiplication C = A @ B with small M and N dimensions (e.g., 64x64) but large K (reduction dimension, e.g., 8192) underutilizes the GPU. Traditional tiling creates few thread blocks (M/64 × N/64 = 1), leaving 99% of SMs idle.

The Solution

Technique: Split the K dimension across multiple blocks and reduce results

Reference: Cutlass split-K GEMM kernels

# Before: Single K dimension (low parallelism)
@triton.jit
def matmul_kernel(A, B, C, M, N, K):
    # Each block computes one output tile: C[bm:bm+64, bn:bn+64]
    # Grid size: (M/64, N/64) = (1, 1) for 64x64 output
    # Only 1 block active → 1/108 SMs utilized on A100
    bm = tl.program_id(0) * 64
    bn = tl.program_id(1) * 64

    acc = tl.zeros((64, 64), dtype=tl.float32)
    for k in range(0, K, 64):
        a = tl.load(A + bm*K + k + offsets_a)
        b = tl.load(B + k*N + bn + offsets_b)
        acc += tl.dot(a, b)
    tl.store(C + bm*N + bn + offsets_c, acc)

# After: Split K across multiple blocks
@triton.jit
def matmul_split_k_kernel(A, B, C_partial, M, N, K, K_SPLIT):
    # Grid size: (M/64, N/64, K_SPLIT) = (1, 1, 32)
    # Now 32 blocks working in parallel!
    bm = tl.program_id(0) * 64
    bn = tl.program_id(1) * 64
    k_split_id = tl.program_id(2)

    # Each block handles a slice of K
    k_start = (K // K_SPLIT) * k_split_id
    k_end = k_start + (K // K_SPLIT)

    acc = tl.zeros((64, 64), dtype=tl.float32)
    for k in range(k_start, k_end, 64):
        a = tl.load(A + bm*K + k + offsets_a)
        b = tl.load(B + k*N + bn + offsets_b)
        acc += tl.dot(a, b)

    # Store partial result
    tl.store(C_partial + k_split_id * M * N + bm*N + bn + offsets_c, acc)

# Separate reduction kernel sums C_partial along K_SPLIT dimension
# Result: 20-30x speedup for large-K, small-MN matrices

The Pattern

✅ When to Apply

  • GEMM with small M, N but large K (tall-skinny or short-wide matrices)
  • Low GPU utilization (few thread blocks)
  • K dimension is much larger than M or N

📋 How to Apply

  • Add K as a grid dimension (split into K_SPLIT slices)
  • Each block computes partial sum over its K slice
  • Run reduction kernel to sum partial results
  • Tune K_SPLIT for occupancy (typically 8-32)

🔍 Profiler Signals

  • Low "Achieved Occupancy" (<10%)
  • Few active SMs despite available compute
  • Matrix dimensions highly unbalanced

Online Softmax Algorithm

The Problem

Naive softmax requires two passes: (1) find max for numerical stability, (2) compute exp and normalize. For long sequences (16K tokens), this means reading the entire logits tensor twice from HBM—doubling memory bandwidth requirements. At 2000 GB/s, a 16K × 16K softmax takes 4ms just for memory access.

The Solution

Technique: Online (streaming) softmax with incremental max update

Reference: Flash Attention paper (Dao et al.), online softmax section

# Before: Two-pass softmax
@triton.jit
def softmax_twopass(logits, output):
    # Pass 1: Find max (full read of logits)
    max_val = float('-inf')
    for i in range(seq_len):
        max_val = max(max_val, logits[i])

    # Pass 2: Compute exp and sum (full read again)
    sum_exp = 0.0
    for i in range(seq_len):
        sum_exp += tl.exp(logits[i] - max_val)

    # Pass 3: Normalize and write
    for i in range(seq_len):
        output[i] = tl.exp(logits[i] - max_val) / sum_exp
# 3 full passes over data

# After: Online softmax (single pass)
@triton.jit
def softmax_online(logits, output):
    # Track running max and sum in registers
    max_val = float('-inf')
    sum_exp = 0.0

    # Single pass: update max and sum incrementally
    for i in range(seq_len):
        old_max = max_val
        max_val = max(max_val, logits[i])

        # Rescale previous sum when max changes
        sum_exp = sum_exp * tl.exp(old_max - max_val) + tl.exp(logits[i] - max_val)

    # Second (smaller) pass for normalization
    for i in range(seq_len):
        output[i] = tl.exp(logits[i] - max_val) / sum_exp
# 2 passes instead of 3, but first pass is fused with other ops in practice
# Result: 1.4x speedup for softmax, enables further fusion

The Pattern

✅ When to Apply

  • Multi-pass reductions over large arrays
  • Statistical operations (mean, variance, softmax)
  • Memory bandwidth is bottleneck

📋 How to Apply

  • Maintain running statistics in registers (max, sum, count)
  • Update statistics incrementally as data streams through
  • Use Welford's algorithm for variance, online softmax for attention

🔍 Profiler Signals

  • Multiple kernel launches for same reduction
  • High HBM traffic for statistical operations
  • Long sequences making two-pass impractical
14

Chunked Prefill

The Problem

Long prompt prefill (e.g., 32K tokens) blocks the GPU for 2-3 seconds, preventing other requests from generating tokens. With 50 concurrent users, this creates 100+ second queuing delays for short requests—terrible user experience. Prefill needs all-to-all attention (O(N²)), dominating GPU time.

The Solution

Technique: Split long prefills into chunks, interleave with decode

Reference: vLLM PR #5316 (Chunked prefill)

# Before: Monolithic prefill
def process_request(prompt_tokens):
    # Compute full prefill in one shot (2+ seconds for 32K tokens)
    kv_cache = model.prefill(prompt_tokens)  # BLOCKS GPU entirely
    # No other requests can make progress

    # Then generate tokens
    for _ in range(max_new_tokens):
        next_token = model.decode(kv_cache)
        kv_cache.append(next_token)

# After: Chunked prefill
def process_request(prompt_tokens, chunk_size=512):
    kv_cache = []

    # Process prompt in chunks
    for i in range(0, len(prompt_tokens), chunk_size):
        chunk = prompt_tokens[i:i+chunk_size]
        kv_chunk = model.prefill(chunk, kv_cache)  # Only 40ms per chunk
        kv_cache.extend(kv_chunk)

        # After each chunk, scheduler can run decode for other requests
        # This interleaving reduces latency for all requests

    # Generate tokens as before
    for _ in range(max_new_tokens):
        next_token = model.decode(kv_cache)
        kv_cache.append(next_token)
# Result: 5-10x reduction in P99 latency for concurrent workloads

The Pattern

✅ When to Apply

  • Long-running operations blocking shared resources
  • Mix of long and short tasks in queue
  • High P99 latency with concurrent load

📋 How to Apply

  • Split work into smaller chunks (512-2048 tokens for prefill)
  • Yield to scheduler between chunks
  • Balance chunk size: too small (overhead), too large (latency)

🔍 Profiler Signals

  • High P99/P95 latency relative to P50
  • GPU fully utilized but poor responsiveness
  • Short tasks queued behind long tasks
15

Continuous Batching

The Problem

Traditional batching waits for all requests in a batch to finish before starting new ones. If one request generates 500 tokens while others finish at 50 tokens, the GPU sits idle for 90% of the time waiting for the long request. Throughput: 20 tokens/sec despite GPU capable of 2000 tokens/sec.

The Solution

Technique: Add new requests to batch as soon as slots free up

Reference: vLLM's core scheduling innovation (Orca-style continuous batching)

# Before: Static batching
def serve_requests(request_queue):
    batch = request_queue[:batch_size]  # Take first N requests
    while any(r.not_finished for r in batch):
        # Run decode step for all requests
        for req in batch:
            if not req.finished:
                token = model.decode(req.kv_cache)
                req.append(token)
        # GPU underutilized if most requests finished early
    # Wait until all done before starting new batch

# After: Continuous batching
def serve_requests(request_queue):
    active_batch = []

    while request_queue or active_batch:
        # Fill empty slots immediately
        while len(active_batch) < batch_size and request_queue:
            active_batch.append(request_queue.pop(0))

        # Run decode for active requests
        for req in active_batch:
            token = model.decode(req.kv_cache)
            req.append(token)

        # Remove finished requests, free slots for new ones
        active_batch = [r for r in active_batch if not r.finished]

        # Loop continues: slots always filled, GPU never idle
# Result: 5-20x throughput improvement, 95%+ GPU utilization

The Pattern

✅ When to Apply

  • Batched workloads with variable completion times
  • GPU idle time due to batch synchronization
  • Queue of pending work always available

📋 How to Apply

  • Track active batch dynamically (not static)
  • Remove finished items immediately
  • Add new items as soon as slots available
  • Requires variable-length batch support (attention masks, KV cache)

🔍 Profiler Signals

  • GPU utilization drops as batch progresses
  • Long tail latency (waiting for slowest request)
  • Throughput far below theoretical peak
16

Speculative Decoding

The Problem

LLM decoding is sequential: generate token 1, then token 2, then token 3. Each token requires a full model forward pass (800ms for Llama-70B). Generating 50 tokens takes 40 seconds. This is inherently serial—no way to parallelize across tokens because token N+1 depends on token N.

The Solution

Technique: Use small draft model to propose multiple tokens, verify in parallel with large model

Reference: Speculative decoding (Leviathan et al. 2023), vLLM speculative decoding implementation

# Before: Serial decoding (slow)
def generate(model, prompt, max_tokens=50):
    kv_cache = model.prefill(prompt)
    output = []
    for _ in range(max_tokens):
        # Each token requires full model forward pass (800ms)
        token = model.decode(kv_cache)  # SEQUENTIAL
        output.append(token)
        kv_cache.append(token)
    return output
# Total time: 50 tokens × 800ms = 40 seconds

# After: Speculative decoding
def generate_speculative(large_model, small_model, prompt, max_tokens=50, k=5):
    kv_cache = large_model.prefill(prompt)
    output = []

    while len(output) < max_tokens:
        # Step 1: Draft model proposes k tokens (fast, 50ms each)
        draft_tokens = []
        draft_kv = kv_cache.clone()
        for _ in range(k):
            token = small_model.decode(draft_kv)  # Small model: 50ms
            draft_tokens.append(token)
            draft_kv.append(token)

        # Step 2: Large model verifies all k tokens IN PARALLEL (800ms total)
        # We can batch-verify because we have the draft sequence
        verified = large_model.verify(kv_cache, draft_tokens)  # Parallel!

        # Step 3: Accept verified prefix
        for i, (draft, correct) in enumerate(zip(draft_tokens, verified)):
            if draft == correct:
                output.append(draft)
                kv_cache.append(draft)
            else:
                output.append(correct)  # Use large model's token
                kv_cache.append(correct)
                break  # Reject remaining draft tokens

# Speedup: If draft accepts 60% of tokens, amortized cost per token:
# (5 × 50ms + 800ms) / 3 accepted = 350ms per token (2.3x speedup!)
# Result: 2-3x decoding speedup with zero quality loss

The Pattern

✅ When to Apply

  • Sequential workloads where dependencies block parallelism
  • Fast approximation available (small model, cached result)
  • Verification can process multiple candidates in parallel

📋 How to Apply

  • Use cheap oracle to propose k candidates
  • Verify all candidates in parallel (batched forward pass)
  • Accept longest correct prefix
  • Tune k and draft quality for acceptance rate

🔍 Profiler Signals

  • Serial execution with no parallelism
  • Each step waits for previous step
  • GPU underutilized (low batch size)
17

Tensor Parallelism

The Problem

Llama-70B has 70 billion parameters (140 GB in FP16), but a single A100 GPU has only 80 GB memory. Model doesn't fit on one GPU. Data parallelism (replicate model across GPUs) doesn't help—you need to split the model itself across multiple GPUs.

The Solution

Technique: Split weight matrices across GPUs, fuse communication with compute

Reference: Megatron-LM tensor parallelism (Shoeybi et al. 2019)

# Problem: Linear layer too large for one GPU
class LinearLayer:
    def __init__(self, in_dim=8192, out_dim=28672):
        self.weight = torch.randn(out_dim, in_dim, dtype=torch.float16)
        # 28672 × 8192 × 2 bytes = 470 MB per layer
        # Llama-70B has 200+ such layers → 94 GB total

    def forward(self, x):  # x: [batch, seq_len, in_dim]
        return x @ self.weight.T  # [batch, seq_len, out_dim]

# Solution: Split weight across 4 GPUs (column-parallel)
class ColumnParallelLinear:
    def __init__(self, in_dim=8192, out_dim=28672, tp_size=4):
        self.tp_rank = torch.distributed.get_rank()
        self.out_dim_per_gpu = out_dim // tp_size  # 28672 / 4 = 7168

        # Each GPU stores 1/4 of weight matrix
        self.weight = torch.randn(self.out_dim_per_gpu, in_dim, dtype=torch.float16)
        # Now only 117 MB per GPU per layer!

    def forward(self, x):
        # All GPUs compute their slice independently (no communication!)
        local_output = x @ self.weight.T  # [batch, seq_len, 7168]
        # Output stays partitioned for next layer
        return local_output

# Next layer must be row-parallel to reduce results
class RowParallelLinear:
    def __init__(self, in_dim=28672, out_dim=8192, tp_size=4):
        self.in_dim_per_gpu = in_dim // tp_size  # 7168
        self.weight = torch.randn(out_dim, self.in_dim_per_gpu, dtype=torch.float16)

    def forward(self, x):  # x is partitioned: [batch, seq_len, 7168]
        # Compute local matmul
        local_output = x @ self.weight.T  # [batch, seq_len, 8192]

        # AllReduce to sum across GPUs (communication happens here)
        torch.distributed.all_reduce(local_output)
        return local_output  # [batch, seq_len, 8192] (reduced)

# Pattern: column-parallel → row-parallel alternate
# Communication only at row-parallel layers (1 AllReduce per 2 layers)
# Result: 70B model fits on 4×80GB GPUs, 3.8x speedup vs sequential

The Pattern

✅ When to Apply

  • Model too large to fit on single GPU memory
  • Want to keep batch size small (inference, not training)
  • Multi-GPU available with fast interconnect (NVLink)

📋 How to Apply

  • Split weight matrices along row or column dimension
  • Alternate: column-parallel (no comm) → row-parallel (AllReduce)
  • Fuse AllReduce with next layer to hide latency
  • Requires custom kernels or framework support (Megatron, vLLM)

🔍 Profiler Signals

  • OOM errors (model doesn't fit)
  • Single GPU underutilized (can't use larger batch)
  • Multi-GPU available but unused

Pipeline Parallelism

The Problem

Large models have many sequential layers (Llama-70B: 80 layers). Running all layers on one GPU is slow (3 seconds per token). Tensor parallelism helps but requires expensive all-reduce after every layer. Alternative: split layers across GPUs, but naive approach leaves GPUs idle waiting for upstream layers (pipeline bubbles).

The Solution

Technique: Pipeline parallelism with microbatching to fill pipeline bubbles

Reference: GPipe (Huang et al. 2019), PipeDream (Narayanan et al. 2019)

# Naive approach: Sequential layer execution across GPUs
def forward_naive(input, layers_per_gpu):
    x = input
    for gpu_id in range(4):
        x = x.to(f'cuda:{gpu_id}')
        for layer in layers_per_gpu[gpu_id]:
            x = layer(x)
    return x
# Problem: GPU 0 busy, GPUs 1-3 idle. Then GPU 1 busy, others idle. Utilization: 25%

# Solution: Pipeline with microbatching
class PipelineParallel:
    def __init__(self, model, num_gpus=4, num_microbatches=8):
        # Split 80 layers across 4 GPUs: 20 layers each
        self.stages = [model.layers[i::num_gpus] for i in range(num_gpus)]
        self.num_microbatches = num_microbatches

    def forward(self, input):
        # Split batch into microbatches
        microbatches = input.chunk(self.num_microbatches)

        # Pipeline schedule (simplified)
        pipeline = []
        for mb in microbatches:
            # Microbatch flows through pipeline stages
            for stage_id, stage_layers in enumerate(self.stages):
                # Wait for input from previous stage (or original input)
                mb_input = pipeline[stage_id - 1] if stage_id > 0 else mb

                # Process on this GPU
                with torch.cuda.device(stage_id):
                    for layer in stage_layers:
                        mb_input = layer(mb_input)

                # Pass to next stage
                pipeline.append(mb_input)

                # KEY: Don't wait for full batch! Next microbatch starts immediately
                # While GPU 0 processes microbatch 2, GPU 1 processes microbatch 1

        # Collect outputs from final stage
        return torch.cat(pipeline[-self.num_microbatches:])

# Timeline visualization:
# Time →   t0      t1      t2      t3      t4
# GPU 0:   mb0     mb1     mb2     mb3     mb4
# GPU 1:   idle    mb0     mb1     mb2     mb3
# GPU 2:   idle    idle    mb0     mb1     mb2
# GPU 3:   idle    idle    idle    mb0     mb1

# After initial "bubble" (t0-t2), all GPUs busy!
# Result: 3.2x speedup with 4 GPUs (80% efficiency after bubble drains)

The Pattern

✅ When to Apply

  • Deep models with many sequential layers
  • Batch size large enough to split (training, offline inference)
  • Multiple GPUs with slower interconnect than NVLink (PCIe)

📋 How to Apply

  • Split model layers across GPUs (stages)
  • Split batch into many microbatches (8-16)
  • Start next microbatch as soon as previous leaves stage
  • Balance stage compute time (profiling-driven layer assignment)

🔍 Profiler Signals

  • GPUs idle waiting for upstream computation
  • Low multi-GPU utilization (<50% aggregate)
  • Model depth > 40 layers

Selective Activation Recomputation

The Problem

Training transformers requires storing all activations for backward pass. For Llama-70B with batch size 32, activations consume 180 GB—more than GPU memory (80 GB). Naive gradient checkpointing saves memory but doubles training time by recomputing everything. Need selective strategy: recompute cheap ops, save expensive ones.

The Solution

Technique: Recompute only activations that are cheap to regenerate

Reference: PyTorch activation checkpointing, Megatron-LM selective recomputation

# Naive: Save all activations (high memory)
class TransformerLayer(nn.Module):
    def forward(self, x):
        # Save ALL intermediate activations for backward
        attn_out = self.attention(x)      # Save: 8 GB
        ln1_out = self.layernorm1(attn_out)  # Save: 8 GB
        mlp_out = self.mlp(ln1_out)       # Save: 32 GB (huge!)
        ln2_out = self.layernorm2(mlp_out)   # Save: 32 GB
        return ln2_out
# Total: 80 GB activations per layer × 80 layers = DISASTER

# Full checkpointing: Save nothing, recompute everything (low memory, slow)
class TransformerLayer(nn.Module):
    def forward(self, x):
        # Don't save any activations
        with torch.no_grad():
            attn_out = self.attention(x)
            ln1_out = self.layernorm1(attn_out)
            mlp_out = self.mlp(ln1_out)
            ln2_out = self.layernorm2(mlp_out)
        return ln2_out
    # Backward: Recompute ALL ops (2x training time)

# Selective recomputation: Save expensive, recompute cheap
class TransformerLayerSelective(nn.Module):
    def forward(self, x):
        # Save input and attention output (expensive ops)
        x_saved = x  # Save: 8 GB
        attn_out = self.attention(x)  # Save: 8 GB (expensive to recompute)

        # Don't save layernorm (cheap to recompute, 0.1ms)
        with torch.no_grad():
            ln1_out = self.layernorm1(attn_out)

        mlp_out = self.mlp(ln1_out)  # Save: 32 GB (expensive)

        with torch.no_grad():
            ln2_out = self.layernorm2(mlp_out)  # Recompute (cheap)

        return ln2_out

    def backward(self, grad):
        # Recompute only cheap ops (layernorms)
        # Use saved expensive ops (attention, MLP)
        pass
# Result: 70% memory reduction, only 15% slowdown (vs 100% for full checkpoint)

The Pattern

✅ When to Apply

  • Training large models with memory constraints
  • Mix of expensive (matmul) and cheap (layernorm, activation) ops
  • Full checkpointing too slow, no checkpointing OOM

📋 How to Apply

  • Profile compute time for each operation type
  • Save activations for expensive ops (attention, MLP, conv)
  • Recompute cheap ops (norm, dropout, GELU, ReLU)
  • Use PyTorch checkpoint or custom autograd.Function

🔍 Profiler Signals

  • OOM during backward pass (activation memory)
  • Training iteration time dominated by memory ops
  • Ratio of cheap:expensive ops > 2:1

Graph-Level Fusion (torch.compile)

The Problem

PyTorch eager mode launches one kernel per operation. A simple GELU activation (`x * 0.5 * (1 + tanh(...))`) launches 8 separate kernels: mul, mul, add, tanh, add, mul, mul—each reading/writing HBM. For transformer with 1000+ operations per layer, this adds 200ms of pure launch and memory overhead.

The Solution

Technique: Trace computation graph, fuse multiple ops into single kernels

Reference: PyTorch 2.0 torch.compile with TorchInductor backend

# Before: Eager mode (many kernels)
def gelu_approx(x):
    # 8 kernel launches, 8 HBM round-trips
    return x * 0.5 * (1.0 + torch.tanh(
        math.sqrt(2.0 / math.pi) * (x + 0.044715 * torch.pow(x, 3.0))
    ))

# Eager execution:
# Kernel 1: pow(x, 3)         → temp1 (write to HBM)
# Kernel 2: mul(temp1, 0.044) → temp2 (write to HBM)
# Kernel 3: add(x, temp2)     → temp3 (write to HBM)
# ... 5 more kernels, each reading/writing HBM
# Total: 200ms (mostly memory, not compute)

# After: torch.compile (fused kernel)
@torch.compile(mode="max-autotune")
def gelu_approx(x):
    # Same Python code, but PyTorch generates ONE fused kernel
    return x * 0.5 * (1.0 + torch.tanh(
        math.sqrt(2.0 / math.pi) * (x + 0.044715 * torch.pow(x, 3.0))
    ))

# Compiled execution:
# Single Triton kernel that:
# - Loads x from HBM once
# - Computes all ops in registers (no intermediate HBM writes!)
# - Writes final result to HBM
# Total: 8ms (12x faster)

# torch.compile also does:
# 1. Operator fusion (combine multiple ops)
# 2. Dead code elimination (remove unused computations)
# 3. Constant folding (precompute constant expressions)
# 4. Layout optimization (NCHW → NHWC if faster)
# 5. Instruction scheduling (hide memory latency)

# Example: Full transformer layer
@torch.compile(mode="max-autotune", fullgraph=True)
class TransformerLayer(nn.Module):
    def forward(self, x):
        # 200+ operations: attention, MLP, layernorm, residual
        # Eager: 200+ kernel launches, 180ms
        # Compiled: 15 fused kernels, 95ms (1.9x speedup)
        ...

The Pattern

✅ When to Apply

  • Many small element-wise operations
  • High kernel launch overhead (>20% of runtime)
  • Static computation graph (same ops each iteration)

📋 How to Apply

  • Use `@torch.compile` decorator (PyTorch 2.0+)
  • Choose mode: "default" (fast compile), "reduce-overhead", "max-autotune" (best perf)
  • Set `fullgraph=True` to force single graph (errors if dynamic control flow)
  • Warm up: first run compiles, subsequent runs use cached kernels

🔍 Profiler Signals

  • Many small kernel launches (<20μs each)
  • High CPU overhead (>15% in PyTorch dispatcher)
  • Timeline shows gaps between kernels
